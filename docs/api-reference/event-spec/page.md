# Page

The Page call lets you record whenever a user sees a page of your website, along with any optional properties about the page. Calling Page is one of the first steps to getting started with RudderStack — many destinations require at least one page event per load for proper initialization.

The Page call is the **web counterpart** to the [Screen](screen.md) call, which serves the same purpose for mobile applications. If your application is mobile-based, use Screen instead.

> **Source references:**
>
> - `gateway/openapi.yaml:135-194` — OpenAPI 3.0.3 endpoint definition for `POST /v1/page`
> - `gateway/handle_http.go:47-49` — `webPageHandler` wires `callType("page", writeKeyAuth(webHandler()))`
> - `refs/segment-docs/src/connections/spec/page.md` — Segment's canonical Page spec reference

> **Segment Compatibility:** RudderStack's Page call is **fully compatible** with the Segment Page spec. All fields, properties, and destination mappings follow identical semantics. Any payload that works with Segment's Page endpoint will work identically with RudderStack — no field-level modifications are required.

> **Important:** In the JavaScript SDK (`analytics.js`), a Page call is **automatically included** in the snippet after `analytics.load`. You can extend this default call with an optional `name` or `properties` object, or call it multiple times per page load for single-page applications (SPAs).

For the common fields shared across all event types (`anonymousId`, `context`, `integrations`, `messageId`, `timestamp`, etc.), see [Common Fields](common-fields.md).

For authentication details required to send events, see the [API Overview & Authentication](../index.md).

## HTTP API

The Page endpoint accepts page view events via HTTP POST.

| Parameter | Value |
|-----------|-------|
| **Method** | `POST` |
| **Path** | `/v1/page` |
| **Port** | `8080` |
| **Content-Type** | `application/json` |
| **Authentication** | Basic Auth with Write Key (username = Write Key, password = empty) |

Source: `gateway/openapi.yaml:135-194`, `gateway/handle_http.go:47-49`

### Authentication

The Page endpoint uses **WriteKey Auth** (HTTP Basic Authentication). The source Write Key is sent as the username, with the password left empty. This is identical to Segment's authentication scheme.

```
Authorization: Basic <base64("YOUR_WRITE_KEY:")>
```

Source: `gateway/openapi.yaml:193-194` — `security: [writeKeyAuth: []]`

### Response Codes

| Status Code | Status | Description | Example Response |
|-------------|--------|-------------|-----------------|
| `200` | OK | Event accepted successfully | `"OK"` |
| `400` | Bad Request | Malformed or invalid payload | `"Invalid request"` |
| `401` | Unauthorized | Missing or invalid Write Key | `"Invalid Authorization Header"` |
| `404` | Not Found | Source not found or does not accept events | `"Source does not accept webhook events"` |
| `413` | Request Entity Too Large | Payload exceeds maximum allowed size | `"Request size too large"` |
| `429` | Too Many Requests | Rate limit exceeded | `"Too many requests"` |

Source: `gateway/openapi.yaml:150-192`

## Fields

The Page call accepts the following top-level fields. Beyond these Page-specific fields, all [Common Fields](common-fields.md) (`anonymousId`, `context`, `integrations`, `messageId`, `timestamp`, `sentAt`, `receivedAt`, etc.) are also accepted.

The `PagePayload` schema is defined at `gateway/openapi.yaml:756-789`.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | String | **Yes** | Must be `"page"`. Identifies this as a Page event. |
| `userId` | String | Optional* | Unique identifier for a particular user in your database. *At least one of `userId` or `anonymousId` is required. |
| `anonymousId` | String | Optional* | Pseudo-unique substitute for a `userId` when there is no unique identifier for the user. Auto-generated by client SDKs. *At least one of `userId` or `anonymousId` is required. |
| `name` | String | Optional | Name of the page being viewed (e.g., `"Home"`, `"Pricing"`, `"Signup"`). |
| `category` | String | Optional | Category of the page (e.g., `"Retail"`, `"Docs"`, `"Blog"`). Useful for grouping pages into high-level sections. |
| `properties` | Object | Optional | Dictionary of properties associated with the page. Includes both reserved properties (`title`, `url`, `path`, `referrer`, `search`, `keywords`) and any custom properties. See the [Properties](#properties) section. |
| `context` | Object | Optional | Dictionary of extra information that provides context about the event (e.g., `ip`, `userAgent`, `library`). See [Common Fields — Context](common-fields.md#context). |
| `integrations` | Object | Optional | Dictionary controlling which destinations receive this event. The special `"All"` key acts as a default. See [Common Fields — Integrations](common-fields.md#integrations). |
| `timestamp` | String (ISO 8601) | Optional | The timestamp of the message's arrival. If not provided, the server sets `receivedAt` automatically. See [Common Fields — Timestamps](common-fields.md#timestamps). |
| `messageId` | String | Optional | Unique identifier for this message. Auto-generated by client SDKs in UUID format. Used by the Gateway's deduplication service to prevent duplicate processing. |

Source: `gateway/openapi.yaml:756-789`

> **Identity Requirement:** Every Page call must include at least one of `userId` or `anonymousId`. If both are provided, the downstream identity resolution process uses `userId` as the primary identifier and associates the `anonymousId` with it.

## Properties

Properties are extra pieces of information that describe the page being viewed. They can be anything you want — from standard web analytics attributes to custom metadata specific to your application.

RudderStack handles properties with **semantic meanings** in unique ways. For example, RudderStack always expects `path` to be a page's URL path and `referrer` to be the previous page's URL. You should **only use reserved properties for their intended meaning**.

### Reserved Properties

The following properties have standardized semantic meaning and are handled specially by RudderStack and downstream destinations:

| Property | Type | Description |
|----------|------|-------------|
| `name` | String | Name of the page. Reserved for future use by destinations. |
| `path` | String | Path portion of the page's URL. Equivalent to `location.pathname` from the DOM API. |
| `referrer` | String | Full URL of the previous page. Equivalent to `document.referrer` from the DOM API. |
| `search` | String | Query string portion of the page's URL. Equivalent to `location.search` from the DOM API. |
| `title` | String | Title of the page. Equivalent to `document.title` from the DOM API. |
| `url` | String | Full URL of the page. The SDK first looks for the canonical URL (`<link rel="canonical">`); if not provided, falls back to `location.href`. |
| `keywords` | Array[String] | A list of keywords describing the page's content. Mainly used by content publishers that rely heavily on pageview tracking. This property is **not** automatically collected. |

Source: `refs/segment-docs/src/connections/spec/page.md:92-102`

> **Auto-Collection:** The JavaScript SDK (`analytics.js`) automatically sends `title`, `path`, `url`, `referrer`, and `search` properties with every Page call. You do not need to manually set these when using the JavaScript SDK on the web.

> **Segment Parity:** All reserved Page properties are handled identically to Segment's Page spec. The property names, types, auto-collection behavior, and semantic handling in destinations are fully compatible.

### Custom Properties

In addition to the reserved properties above, you can pass any number of custom key-value pairs in the `properties` object. Custom properties are forwarded to all downstream destinations that accept them.

```json
{
  "properties": {
    "title": "Welcome | Initech",
    "url": "http://www.example.com",
    "path": "/",
    "author": "Peter Gibbons",
    "publishDate": "2025-01-15",
    "wordCount": 1250
  }
}
```

## Examples

### Minimal Page Call

The simplest valid Page call includes the event `type`, a `name`, and at least one identity field (`userId` or `anonymousId`):

```json
{
  "type": "page",
  "name": "Home",
  "anonymousId": "507f191e810c19729de860ea",
  "properties": {
    "title": "Welcome | Initech",
    "url": "http://www.example.com"
  }
}
```

### curl Example

Send a Page event to the RudderStack Gateway using `curl`. The Write Key is passed as the HTTP Basic Auth username (note the trailing colon indicating an empty password):

```bash
curl -X POST http://localhost:8080/v1/page \
  -u "YOUR_WRITE_KEY:" \
  -H "Content-Type: application/json" \
  -d '{
    "userId": "user123",
    "name": "Home",
    "properties": {
      "title": "Welcome | Initech",
      "url": "http://www.example.com",
      "path": "/",
      "referrer": "https://google.com"
    }
  }'
```

A successful response returns HTTP `200` with body `OK`.

### JavaScript SDK Examples

The RudderStack JavaScript SDK provides a convenient `analytics.page()` method that automatically collects page properties (`title`, `path`, `url`, `referrer`, `search`) from the DOM:

```javascript
// Basic page call — auto-collects name and properties from the DOM
analytics.page();

// Page call with an explicit name
analytics.page("Home");

// Page call with category and name
analytics.page("Retail", "Home");

// Page call with name and custom properties
analytics.page("Home", {
  title: "Welcome | Initech",
  url: "http://www.example.com"
});

// Page call with category, name, and properties
analytics.page("Docs", "Getting Started", {
  title: "Getting Started Guide",
  url: "http://www.example.com/docs/getting-started",
  author: "Engineering Team"
});
```

> **Note:** A Page call is automatically included in the default JavaScript SDK snippet after `analytics.load()`. For single-page applications (SPAs), call `analytics.page()` manually on each virtual page navigation.

### Server-Side Example (Node.js)

```javascript
analytics.page({
  userId: "user123",
  category: "Docs",
  name: "Getting Started",
  properties: {
    title: "Getting Started Guide",
    url: "http://www.example.com/docs/getting-started",
    path: "/docs/getting-started",
    referrer: "http://www.example.com/docs"
  },
  context: {
    ip: "8.8.8.8"
  },
  timestamp: new Date().toISOString()
});
```

## Segment Behavioral Parity

RudderStack's Page call achieves **full behavioral parity** with the Segment Page spec. The following table provides a field-by-field comparison to confirm identical semantics across both platforms.

| Field | Segment Behavior | RudderStack Behavior | Parity Status |
|-------|-----------------|---------------------|---------------|
| `type` | Must be `"page"` | Must be `"page"` | ✅ Full Parity |
| `name` | Optional page name string | Optional page name string | ✅ Full Parity |
| `category` | Optional page category string | Optional page category string | ✅ Full Parity |
| `userId` | Optional (one of userId/anonymousId required) | Optional (one of userId/anonymousId required) | ✅ Full Parity |
| `anonymousId` | Optional (one of userId/anonymousId required) | Optional (one of userId/anonymousId required) | ✅ Full Parity |
| `properties` | Free-form object with reserved keys | Free-form object with reserved keys | ✅ Full Parity |
| `properties.title` | Auto-collected from `document.title` | Auto-collected by JS SDK from `document.title` | ✅ Full Parity |
| `properties.url` | Auto-collected, canonical URL preference | Auto-collected by JS SDK, canonical URL preference | ✅ Full Parity |
| `properties.path` | Auto-collected from `location.pathname` | Auto-collected by JS SDK from `location.pathname` | ✅ Full Parity |
| `properties.referrer` | Auto-collected from `document.referrer` | Auto-collected by JS SDK from `document.referrer` | ✅ Full Parity |
| `properties.search` | Auto-collected from `location.search` | Auto-collected by JS SDK from `location.search` | ✅ Full Parity |
| `properties.keywords` | Manual — not auto-collected | Manual — not auto-collected | ✅ Full Parity |
| `context` | Common context object | Common context object (identical schema) | ✅ Full Parity |
| `integrations` | Destination routing dictionary | Destination routing dictionary (identical schema) | ✅ Full Parity |
| `timestamp` | ISO 8601, optional | ISO 8601, optional | ✅ Full Parity |
| `messageId` | UUID, auto-generated by SDKs | UUID, auto-generated by SDKs | ✅ Full Parity |
| Auto-page on load | `analytics.js` includes Page in snippet | JS SDK includes Page call after `analytics.load` | ✅ Full Parity |

Source: `refs/segment-docs/src/connections/spec/page.md`, `gateway/openapi.yaml:756-789`

**Migration Note:** When migrating from Segment to RudderStack, Page calls require **zero payload modifications**. Simply update the `dataPlaneUrl` configuration in your SDK initialization to point at your RudderStack Gateway (port 8080), and all Page events will be processed identically.
