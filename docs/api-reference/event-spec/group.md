# Group

The **Group** API call associates an individual user with a group — such as a company, organization, account, project, or team. It enables you to identify which account or organization your users belong to, and to record custom traits about that group (for example, company name, industry, employee count, or subscription plan).

Two IDs are relevant in a Group call:

- **`userId`** — belongs to and identifies the individual user.
- **`groupId`** — belongs to and identifies the specific group.

A single user can be associated with multiple groups (different `groupId` values), but the user retains only one `userId` across all of those groups. Keep in mind that not all downstream platforms support multiple groups for a single user.

> **Source references:**
>
> - `gateway/openapi.yaml:256-317` — endpoint definition for `POST /v1/group`
> - `gateway/handle_http.go:67-69` — `webGroupHandler` wires `callType("group", writeKeyAuth(webHandler()))`
> - `refs/segment-docs/src/connections/spec/group.md` — Segment's canonical Group specification

> **Segment Compatibility:** RudderStack's Group call is **fully compatible** with the Segment Group spec. Any payload that works with Segment's Group endpoint will work identically with RudderStack — no field-level modifications are required.

> **Note:** RudderStack does not have a built-in method for removing users from a group (matching Segment behavior). For device-mode destinations that support group removal, invoke the removal method directly on the client side using the `ready()` callback. For cloud-mode destinations, use [Destination Functions](../../guides/transformations/functions.md) to implement group removal logic.

The Group call is especially helpful for B2B applications where accounts contain multiple users.

For the fields shared across all event types (timestamps, context, integrations, etc.), see [Common Fields](common-fields.md).

## Fields

Beyond the [common fields](common-fields.md) shared by all event types, the Group call accepts the following fields:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | String | **Yes** | Must be `"group"`. |
| `userId` | String | Optional\* | Unique identifier for the user in your database. |
| `anonymousId` | String | Optional\* | Pseudo-unique substitute identifier for anonymous users. |
| `groupId` | String | **Yes** | Unique identifier of the group as present in your database. |
| `traits` | Object | Optional | Dictionary of traits associated with the group (e.g., name, industry, employees, plan). |
| `context` | Object | Optional | Dictionary of extra information providing context about the message (IP, user agent, library, etc.). See [Common Fields — Context](common-fields.md#context). |
| `timestamp` | String (ISO 8601) | Optional | Timestamp of the message's original occurrence. |
| `integrations` | Object | Optional | Dictionary of destinations to enable or disable for this event. See [Common Fields — Integrations](common-fields.md#integrations). |
| `messageId` | String | Optional | Unique identifier for this message. Auto-generated by SDKs if not provided. |

> **\* Identity Requirement:** Every Group call must include at least one of `userId` or `anonymousId`. Requests containing neither will be rejected with HTTP `400`.

> **`groupId`** is the unique identifier your organization uses for the group — typically a database primary key or a MongoDB-style ObjectId (e.g., `"0e8c78ea9d97a7b8185e8632"`).

Source: `gateway/openapi.yaml:826-867` — `GroupPayload` schema

## Group ID

A Group ID is the unique identifier by which you recognize a group in your own database. For example, if you are using MongoDB, your Group ID might look like `507f191e810c19729de860ea`. If you use an auto-incrementing integer, it might simply be `42`.

The `groupId` is required on every Group call and is the primary key used by downstream destinations to create or update group records.

Source: `refs/segment-docs/src/connections/spec/group.md:107-108`

## Traits

Traits are pieces of information you know about a group that are passed along with the Group call — for example, `employees`, `industry`, `plan`, or `website`.

In addition to the `groupId`, which identifies the specific group, the Group method accepts traits that are specific to the group. Like the traits of an Identify call, you can update group traits by sending the same trait key with a different value. Traits follow **NoSQL-style upsert semantics**:

- **Sending a trait with a new value** overwrites the previous value for that key.
- **Omitting a previously-sent trait** preserves its prior value — it is **not** deleted.
- **Sending a trait with a `null` value** replaces the stored value with `null`.

> ⚠️ **Warning:** Sending a reserved trait with a `null` value may cause issues in some downstream destinations that expect non-null values for those fields.

### Reserved Group Traits

RudderStack (like Segment) reserves certain trait names with semantic meanings for groups. You should **only use reserved traits for their intended purpose**.

| Trait | Type | Description |
|-------|------|-------------|
| `address` | Object | Street address of the group. Should be a dictionary containing optional `city`, `country`, `postalCode`, `state`, or `street`. |
| `avatar` | String | URL to an avatar image for the group. |
| `createdAt` | Date | Date the group's account was first created. Use [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) date strings. |
| `description` | String | Description of the group (e.g., company bio or mission statement). |
| `email` | String | Email address of the group (e.g., a team or company contact address). |
| `employees` | String | Number of employees of the group, typically used for companies. |
| `id` | String | Unique ID in your database for the group. |
| `industry` | String | Industry the group belongs to (e.g., "Technology", "Finance", "Healthcare"). |
| `name` | String | Name of the group (e.g., company name). |
| `phone` | String | Phone number of the group. |
| `website` | String | Website URL of the group. |
| `plan` | String | Plan that the group is on (e.g., "enterprise", "free", "pro"). |

Source: `refs/segment-docs/src/connections/spec/group.md:117-132`

> **Note:** RudderStack automatically handles destination-specific conversions for reserved traits. For example, Mixpanel's special `track_charges` method for accepting revenue is called automatically — you do not need to handle these conversions yourself.

> **Traits are case-insensitive.** In JavaScript you can match your camelCase code by sending `createdAt`, and in Ruby you can match your snake_case code by sending `created_at`. The API accepts both forms seamlessly.

**Segment Parity:** All 12 reserved Group traits are handled identically to Segment. Reserved trait names, types, and semantic mappings are fully compatible.

## Examples

### Minimal Group Payload

A Group call with most [common fields](common-fields.md) removed, showing only the Group-specific fields:

```json
{
  "type": "group",
  "groupId": "0e8c78ea9d97a7b8185e8632",
  "traits": {
    "name": "Initech",
    "industry": "Technology",
    "employees": 329,
    "plan": "enterprise",
    "total billed": 830
  }
}
```

### curl Example

```bash
curl -X POST http://localhost:8080/v1/group \
  -u "YOUR_WRITE_KEY:" \
  -H "Content-Type: application/json" \
  -d '{
    "userId": "user123",
    "groupId": "0e8c78ea9d97a7b8185e8632",
    "traits": {
      "name": "Initech",
      "industry": "Technology",
      "employees": 329,
      "plan": "enterprise"
    }
  }'
```

> **Note:** Replace `YOUR_WRITE_KEY` with your actual Source Write Key. The trailing colon after the Write Key in the `-u` flag indicates an empty password for HTTP Basic Auth.

### JavaScript SDK Example

```javascript
analytics.group("0e8c78ea9d97a7b8185e8632", {

  industry: "Technology",
  employees: 329,
  plan: "enterprise",
  "total billed": 830
});
```


## Identities

Every Group call must include at least one identity field:

| Field | Description |
|-------|-------------|
| `userId` | The permanent database-level unique identifier for the user. Should persist across sessions. Preferred over `anonymousId` when the user is known. |
| `anonymousId` | A pseudo-unique substitute for a `userId` when the user is not yet identified. Generated automatically by client SDKs (typically a UUIDv4). |

Both fields can be sent together. If only `anonymousId` is provided, downstream destinations will associate the group membership with the anonymous profile until an Identify call links the anonymous ID to a known user.

## Segment Behavioral Parity

Per the Segment Spec behavioral parity requirement (AAP §0.10), this section documents field-by-field compatibility between the Segment Group call and the RudderStack Group call.

| Feature | Segment Behavior | RudderStack Behavior | Parity Status |
|---------|-----------------|---------------------|---------------|
| `groupId` | Required group identifier | Required group identifier | ✅ Full Parity |
| `traits` | Group attributes with semantic mapping | Group attributes with semantic mapping | ✅ Full Parity |
| Multi-group support | User can be associated with multiple groups | User can be associated with multiple groups | ✅ Full Parity |
| Trait overwrites | NoSQL-style upsert — new value overwrites previous | NoSQL-style upsert — new value overwrites previous | ✅ Full Parity |
| Trait preservation | Omitted traits retain prior values | Omitted traits retain prior values | ✅ Full Parity |
| Reserved traits | 12 standardized traits (address, avatar, createdAt, description, email, employees, id, industry, name, phone, website, plan) | 12 standardized traits — identical set | ✅ Full Parity |
| Trait case-insensitivity | Case-insensitive trait keys | Case-insensitive trait keys | ✅ Full Parity |
| No native group removal | No built-in removal method | No built-in removal method | ✅ Full Parity |
| `userId`/`anonymousId` | At least one required | At least one required | ✅ Full Parity |
| `context` | Standard context dictionary | Standard context dictionary | ✅ Full Parity |
| `integrations` | Destination routing dictionary | Destination routing dictionary | ✅ Full Parity |
| `timestamp` | ISO 8601, computed as `receivedAt - (sentAt - originalTimestamp)` | ISO 8601, same computation | ✅ Full Parity |
| `messageId` | UUID auto-generation, dedup key | UUID auto-generation, dedup key | ✅ Full Parity |

**Summary:** The RudderStack Group call achieves **100% behavioral parity** with the Segment Group call. Payloads accepted by Segment's `/v1/group` endpoint are accepted identically by RudderStack's `/v1/group` endpoint with no field-level modifications required.
