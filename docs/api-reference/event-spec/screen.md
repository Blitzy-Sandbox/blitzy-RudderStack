# Screen

The **Screen** call lets you record whenever a user sees a screen in your mobile app, along with any optional properties about the screen. It is the **mobile equivalent** of the [Page](page.md) call for web applications.

Screen calls are the foundation of mobile analytics — they allow you to track user navigation patterns, measure screen-level engagement, and build conversion funnels across your mobile app's user interface.

> **Source references:**
>
> - `gateway/openapi.yaml:195-255` — OpenAPI endpoint definition for `POST /v1/screen`
> - `gateway/handle_http.go:52-54` — `webScreenHandler` wires `callType("screen", writeKeyAuth(webHandler()))`
> - `refs/segment-docs/src/connections/spec/screen.md` — Segment Screen spec reference

> **Segment Behavioral Parity:** RudderStack's Screen call is **fully compatible** with the Segment Screen specification. Any payload that works with Segment's Screen endpoint will work identically with RudderStack — no field-level modifications are required.

For details on the shared fields present in all event types (such as `context`, `integrations`, `anonymousId`, `userId`, `timestamp`, and `messageId`), see the [Common Fields](common-fields.md) reference.

## HTTP API

### Endpoint

| Property | Value |
|----------|-------|
| **Method** | `POST` |
| **Path** | `/v1/screen` |
| **Port** | `8080` (Gateway default) |
| **Content-Type** | `application/json` |
| **Authentication** | Basic Auth with Write Key |

**Authentication:** The Screen endpoint uses `writeKeyAuth` — HTTP Basic Authentication where the username is your source **Write Key** and the password is an empty string. This is identical to Segment's authentication scheme.

Source: `gateway/openapi.yaml:254-255` — `security: [writeKeyAuth: []]`

```
Authorization: Basic <base64(YOUR_WRITE_KEY:)>
```

For the full authentication guide covering all five auth schemes, see [API Overview & Authentication](../index.md).

### Response Codes

| HTTP Status | Status Name | Description | Example Response |
|-------------|-------------|-------------|-----------------|
| **200** | OK | Event accepted successfully | `"OK"` |
| **400** | Bad Request | Malformed or invalid payload | `"Invalid request"` |
| **401** | Unauthorized | Missing or invalid Write Key | `"Invalid Authorization Header"` |
| **404** | Not Found | Source not configured or does not accept events | `"Source does not accept webhook events"` |
| **413** | Request Entity Too Large | Payload exceeds maximum allowed size | `"Request size too large"` |
| **429** | Too Many Requests | Rate limit exceeded | `"Too many requests"` |

Source: `gateway/openapi.yaml:210-255`

## Fields

The Screen call accepts the following fields. The `ScreenPayload` schema is defined at `gateway/openapi.yaml:790-825`.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | String | **Yes** | Must be `"screen"`. Identifies this as a Screen event. |
| `userId` | String | Optional* | Unique identifier for a particular user in your database. Source: `gateway/openapi.yaml:793-795` |
| `anonymousId` | String | Optional* | Sets the user ID for cases where there is no unique identifier for the user. At least one of `userId` or `anonymousId` must be provided. Source: `gateway/openapi.yaml:796-800` |
| `name` | String | Optional | Name of the screen being viewed (e.g., `"Home"`, `"Signup"`, `"Settings"`). Source: `gateway/openapi.yaml:801-803` |
| `properties` | Object | Optional | Free-form dictionary of properties associated with the screen. Source: `gateway/openapi.yaml:804-808` |
| `context` | Object | Optional | Dictionary of extra context about the message. Often includes mobile-specific context such as `device`, `os`, `app`, and `screen` dimensions. Source: `gateway/openapi.yaml:809-821` |
| `timestamp` | String (ISO 8601) | Optional | Timestamp when the message was originally sent, in ISO 8601 format (e.g., `"2024-01-15T10:30:00.000Z"`). Source: `gateway/openapi.yaml:822-825` |
| `integrations` | Object | Optional | Dictionary that controls which destinations receive this event. Keys are destination names; values are booleans. See [Common Fields](common-fields.md#integrations). |
| `messageId` | String | Optional | Unique identifier for this message. Auto-generated by SDKs if not provided. |
| `channel` | String | Optional | Channel the event originated from. Typically `"mobile"` for Screen events. |

*\*At least one of `userId` or `anonymousId` is required for user identification.*

## Screen Name

Each screen can be tagged with a `name` that identifies the screen the user is viewing. For example, many apps have a `"Home"` screen, a `"Signup"` screen, or a `"Settings"` screen that can be useful to label so you can track users as they move through your app's navigation flow.

The `name` field is **optional** but strongly recommended — it enables:

- **Funnel analysis:** Track how users progress through multi-step flows (e.g., Onboarding → Signup → Verification → Home).
- **Screen-level engagement:** Measure time-on-screen and interaction rates for specific screens.
- **Cohort segmentation:** Group users by the screens they visit.

```json
{
  "type": "screen",
  "name": "Signup",
  "userId": "user_abc123"
}
```

Source: `refs/segment-docs/src/connections/spec/screen.md:80-82`

## Properties

Properties are extra pieces of information that describe the screen. They can be anything you want — free-form key/value pairs that add context to the screen view.

```json
{
  "type": "screen",
  "name": "Home",
  "properties": {
    "Feed Type": "private",
    "theme": "dark",
    "ab_variant": "B"
  }
}
```

### Reserved Properties

RudderStack has reserved some properties with semantic meanings and handles them in special ways. You should **only use reserved properties for their intended meaning**.

| Property | Type | Description |
|----------|------|-------------|
| `name` | String | Name of the screen. This is reserved for future use. |

Source: `refs/segment-docs/src/connections/spec/screen.md:89-95`

> **Segment Parity:** Screen properties are handled identically to Segment. Free-form properties are passed through to all enabled destinations without modification, and reserved properties retain the same semantic meaning.

## Examples

### Minimal Screen Call

The simplest Screen call requires only the `type` and at least one user identifier (`userId` or `anonymousId`):

```json
{
  "type": "screen",
  "name": "Home",
  "properties": {
    "Feed Type": "private"
  }
}
```

### curl Example

Send a Screen event directly to the RudderStack Gateway using `curl`:

```bash
curl -X POST http://localhost:8080/v1/screen \
  -u "YOUR_WRITE_KEY:" \
  -H "Content-Type: application/json" \
  -d '{
    "userId": "user123",
    "name": "Home",
    "properties": {
      "Feed Type": "private"
    },
    "context": {
      "device": {
        "type": "ios",
        "manufacturer": "Apple",
        "model": "iPhone14,2"
      },
      "os": {
        "name": "iOS",
        "version": "16.0"
      }
    }
  }'
```

> **Note:** Replace `YOUR_WRITE_KEY` with your actual source Write Key. The trailing colon after the Write Key is required (it serves as the empty password in HTTP Basic Auth). The Gateway listens on port **8080** by default.

### iOS SDK Example (Objective-C)

```objc
[[SEGAnalytics sharedAnalytics] screen:@"Home"
                            properties:@{ @"Feed Type": @"private" }];
```

### Android SDK Example (Kotlin)

```kotlin
analytics.screen("Home", properties = buildJsonObject {
    put("Feed Type", "private")
})
```

### Screen Call with Rich Mobile Context

A more realistic mobile Screen call includes device, OS, app, and screen metadata in the `context` object:

```json
{
  "type": "screen",
  "userId": "user_abc123",
  "name": "Settings",
  "properties": {
    "section": "Account",
    "dark_mode": true
  },
  "context": {
    "app": {
      "name": "MyApp",
      "version": "3.2.1",
      "build": "421",
      "namespace": "com.example.myapp"
    },
    "device": {
      "id": "B5372DB0-C21E-11E4-8DFC-AA07A5B093DB",
      "manufacturer": "Apple",
      "model": "iPhone14,2",
      "name": "iPhone 13 Pro",
      "type": "ios"
    },
    "os": {
      "name": "iOS",
      "version": "16.4.1"
    },
    "screen": {
      "width": 390,
      "height": 844,
      "density": 3
    },
    "network": {
      "wifi": true,
      "cellular": false,
      "carrier": "T-Mobile"
    },
    "locale": "en-US",
    "timezone": "America/New_York",
    "library": {
      "name": "rudder-ios-sdk",
      "version": "2.4.0"
    }
  },
  "integrations": {
    "All": true,
    "Amplitude": true,
    "Mixpanel": false
  },
  "timestamp": "2024-01-15T14:32:00.000Z",
  "messageId": "a8c1f6e0-b234-11e4-ab27-0800200c9a66"
}
```

> **Tip:** Mobile SDKs automatically populate the `context.device`, `context.os`, `context.app`, `context.screen`, and `context.network` fields. You typically only need to provide `name` and `properties` in your SDK calls.

## Segment Behavioral Parity

RudderStack's Screen call maintains **full behavioral parity** with the Segment Screen specification. The following field-by-field comparison confirms identical behavior across both platforms (per AAP §0.10 — Segment behavioral parity is the acceptance criterion).

| Field | Segment Behavior | RudderStack Behavior | Parity Status |
|-------|-----------------|---------------------|---------------|
| `type` | Must be `"screen"` | Must be `"screen"` | ✅ Full Parity |
| `name` | Optional screen name | Optional screen name | ✅ Full Parity |
| `properties` | Free-form metadata dictionary | Free-form metadata dictionary | ✅ Full Parity |
| `userId` | Optional user identifier | Optional user identifier | ✅ Full Parity |
| `anonymousId` | Optional anonymous identifier | Optional anonymous identifier | ✅ Full Parity |
| `channel` | Typically `"mobile"` | Typically `"mobile"` | ✅ Full Parity |
| `context` | Auto-collected context object | Auto-collected context object | ✅ Full Parity |
| `context.device` | Auto-collected by mobile SDKs | Auto-collected by mobile SDKs | ✅ Full Parity |
| `context.os` | Auto-collected by mobile SDKs | Auto-collected by mobile SDKs | ✅ Full Parity |
| `context.app` | Auto-collected by mobile SDKs | Auto-collected by mobile SDKs | ✅ Full Parity |
| `context.screen` | Auto-collected screen dimensions | Auto-collected screen dimensions | ✅ Full Parity |
| `integrations` | Destination-level routing control | Destination-level routing control | ✅ Full Parity |
| `timestamp` | ISO 8601 timestamp | ISO 8601 timestamp | ✅ Full Parity |
| `messageId` | Auto-generated unique ID | Auto-generated unique ID | ✅ Full Parity |

> **Migration Note:** If you are migrating from Segment, your existing Screen calls require **zero payload modifications**. Simply update your SDK's `dataPlaneUrl` (or API endpoint) to point at your RudderStack Gateway on port 8080. All Screen payloads will be processed identically.
