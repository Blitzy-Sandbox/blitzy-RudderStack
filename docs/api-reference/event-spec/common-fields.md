# Common Fields

Every API call in RudderStack shares a common structure and a set of fields. These fields describe user identity, timestamps, context metadata, and mechanical aides like API version. This common structure applies to all six core event types: [Identify](identify.md), [Track](track.md), [Page](page.md), [Screen](screen.md), [Group](group.md), and [Alias](alias.md).

This structure is **fully compatible** with the Segment Spec common fields specification. Any payload that works with Segment's HTTP API will work identically with RudderStack — no field-level modifications are required.

> **Source references:**
>
> - `refs/segment-docs/src/connections/spec/common.md` — Segment's canonical common fields reference
> - `gateway/openapi.yaml:687-939` — OpenAPI 3.0.3 schema definitions showing common fields across all payload types
> - `gateway/types.go:19-31` — `webRequestT` struct showing internal request field handling

> **Note:** Not all destinations accept all common fields. Refer to destination-specific documentation for field support details. See the [Destination Catalog](../../guides/destinations/index.md) for per-destination field matrices.

For authentication details required to send events to these endpoints, see the [API Overview & Authentication](../index.md).

### Common Fields Reference

The following table documents every common field present across all API call types:

| Field | Type | Description |
|-------|------|-------------|
| `anonymousId` | String | Pseudo-unique substitute for a `userId` when no permanent identifier is available. Generated automatically by client SDKs using UUIDv4. At least one of `anonymousId` or `userId` must be present in every call. |
| `context` | Object | Dictionary of extra information that provides useful context about a datapoint — for example, the user's `ip` address or `locale`. See the [Context](#context) section below for the full field list. |
| `integrations` | Object | Dictionary controlling which destinations receive this event. The special `"All"` key acts as a default. See the [Integrations](#integrations) section below. |
| `messageId` | String | Unique identifier for this message. Auto-generated by client SDKs in UUID format. Used by the Gateway's deduplication service to prevent duplicate processing. |
| `receivedAt` | String (ISO 8601) | Server-side timestamp set by RudderStack when the event is received. Used as the primary sort key in warehouse destinations. See [Timestamps](#timestamps). |
| `sentAt` | String (ISO 8601) | Client-side timestamp set when the event batch was dispatched to RudderStack. Used in combination with `receivedAt` to correct for client clock skew. See [Timestamps](#timestamps). |
| `timestamp` | String (ISO 8601) | Corrected event timestamp, computed as `receivedAt - (sentAt - originalTimestamp)`. This is the canonical event time delivered to all downstream destinations. Can be explicitly set when using server-side libraries. See [Timestamps](#timestamps). |
| `type` | String | The event type. One of: `"identify"`, `"track"`, `"page"`, `"screen"`, `"group"`, or `"alias"`. |
| `userId` | String | Permanent, database-level unique identifier for the user. Should persist across sessions and devices. At least one of `userId` or `anonymousId` must be present. |
| `version` | Number | API version number. Currently `2`. |

Source: `refs/segment-docs/src/connections/spec/common.md:115-129`, `gateway/openapi.yaml:688-939`

Beyond this common structure, each API call adds specialized top-level fields. See individual event spec documentation:

- [Identify](identify.md) adds `traits` (user attributes)
- [Track](track.md) adds `event` (event name) and `properties` (event metadata)
- [Page](page.md) adds `name` (page name) and `properties` (page attributes)
- [Screen](screen.md) adds `name` (screen name) and `properties` (screen attributes)
- [Group](group.md) adds `groupId` (group identifier) and `traits` (group attributes)
- [Alias](alias.md) adds `previousId` (previous user identifier)

RudderStack extends the common field set with the following additional fields:

|-------|------|-------------|
| `originalTimestamp` | String (ISO 8601) | Client-side timestamp of when the event was originally invoked. Used in the `timestamp` computation formula: `receivedAt - (sentAt - originalTimestamp)`. Not useful for direct analysis due to client clock skew. |
| `channel` | String | Indicates where the request originated. One of: `"server"`, `"browser"`, or `"mobile"`. Automatically set by client SDKs. |

## Context

Context is a dictionary of extra information that provides useful context about a datapoint — for example, the user's `ip` address or `locale`. Context fields should **only** be used for their intended meaning as documented below.

The context dictionary supports 18 standard sub-fields, all of which match the Segment Spec context specification identically.

Source: `refs/segment-docs/src/connections/spec/common.md:133-156`

| Field | Type | Description |
|-------|------|-------------|
| `active` | Boolean | Whether a user is active. Usually used to flag an Identify call to update traits but not "last seen." |
| `app` | Object | Current application info: `name`, `version`, `build`, `namespace`. Auto-collected from mobile libraries. |
| `campaign` | Object | Campaign attribution info: `name`, `source`, `medium`, `term`, `content`, and any custom UTM parameters. Maps directly to standard UTM campaign parameters. |
| `device` | Object | Device info: `id`, `advertisingId`, `adTrackingEnabled`, `manufacturer`, `model`, `name`, `type`, `version`, `token`. Note: on iOS devices, the `model` value set by Apple might not correspond exactly to the marketed model name (e.g., iPhone 15 Pro Max reports as `iPhone16,2`). |
| `ip` | String | Current user's IP address. For client-side events, this is filled in server-side by RudderStack on receipt — it is not collected by client libraries. |
| `library` | Object | Library making the API request: `name` and `version`. Auto-collected by all SDKs. |
| `locale` | String | Locale string for the current user (e.g., `"en-US"`). |
| `network` | Object | Current network connection info: `bluetooth`, `carrier`, `cellular`, `wifi`. If both `cellular` and `wifi` are empty, the user is offline. |
| `os` | Object | Operating system info: `name` and `version`. |
| `page` | Object | Current browser page info: `path`, `referrer`, `search`, `title`, `url`. Auto-collected by Analytics.js on web. |
| `referrer` | Object | Referral source info: `type`, `name`, `url`, `link`. |
| `screen` | Object | Device screen info: `density`, `height`, `width`. |
| `timezone` | String | User timezone in [tzdata](https://en.wikipedia.org/wiki/Tz_database) format (e.g., `"America/New_York"`, `"Europe/Amsterdam"`). May be unavailable due to browser limitations, privacy settings, or missing API support. |
| `groupId` | String | Group / Account ID. Useful in B2B use cases where you need to attribute non-group calls to a company or account. Relied on by several Customer Success and CRM tools. |
| `traits` | Object | Traits of the current user. Useful for passing previously-identified user traits alongside Track calls via `context.traits`. Populate this the same way you would fill traits in an [Identify](identify.md) call. |
| `userAgent` | String | User agent of the device making the request. |
| `userAgentData` | Object | Client Hints API data. Always contains `brands`, `mobile`, `platform`. May additionally contain `bitness`, `model`, `platformVersion`, `uaFullVersion`, `fullVersionList`, `wow64` if requested and available. Populates only when the [Client Hints API](https://developer.mozilla.org/en-US/docs/Web/API/User-Agent_Client_Hints_API) is available in the browser. May provide more detail than `userAgent` in some cases. |
| `channel` | String | Where the request originated: `"server"`, `"browser"`, or `"mobile"`. |

### Manually Sending Context Fields

To pass context variables that are not automatically collected, include them in the event payload manually. For example, passing `groupId` in a Track call using the JavaScript SDK:

```javascript
analytics.track("Report Submitted", {}, {
  context: {
    groupId: "1234"
  }
});
```

For server-side libraries, include the full context object:

```javascript
// Node.js SDK example
analytics.track({
  userId: "97980cfea0067",
  event: "Report Submitted",
  context: {
    ip: "8.8.8.8",
    locale: "en-US",
    timezone: "America/New_York",
    userAgent: "Mozilla/5.0 ..."
  }
});
```

## Integrations

The `integrations` dictionary is a top-level field that controls which destinations receive this event. It is a dictionary of destination names mapped to boolean values.

The special `"All"` key acts as a default fallback:

- `"All": true` — send to all enabled destinations (default)
- `"All": false` — disable all destinations, then use individual keys to opt in

Individual destination keys override the `"All"` setting. For example, to send to all destinations except Mixpanel and Salesforce:

```json
{
  "integrations": {
    "All": true,
    "Mixpanel": false,
    "Salesforce": false
  }
}
```

### Default Behavior

The integrations dictionary defaults to:

```json
{
  "All": true,
  "Salesforce": false
}
```

Salesforce is disabled by default because it has strict rate limits on API calls. All other destinations are opt-out — if you don't explicitly set a destination to `false`, it will receive the event if it is configured and can accept the event type.

Source: `refs/segment-docs/src/connections/spec/common.md:231-246`

### Selective Opt-In Example

To send events only to specific destinations, set `"All": false` and enable individual destinations:

```json
{
  "integrations": {
    "All": false,
    "Google Analytics": true,
    "Amplitude": true
  }
}
```

**Segment Parity:** The `integrations` dictionary follows identical semantics to Segment. Destination opt-in/opt-out logic, the `"All"` key behavior, and the Salesforce default exclusion are fully compatible. No changes are required when migrating from Segment.

## Timestamps

Every API call carries four timestamps that serve different purposes in the event pipeline. Understanding timestamp semantics is critical for correct event ordering, clock-skew correction, and warehouse query optimization.

**All timestamps are [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) date strings in UTC timezone.** To access user timezone information, use the `context.timezone` field which is [automatically collected](#context-fields-automatically-collected) by client-side libraries.

Source: `refs/segment-docs/src/connections/spec/common.md:249-299`

### Timestamp Overview

| Timestamp | Set By | Description | Use Case |
|-----------|--------|-------------|----------|
| `originalTimestamp` | Client SDK | Time the event was invoked on the client device. **OR** the `timestamp` value manually passed in through server-side libraries. | Used in clock-skew correction calculation. **Not recommended** for direct analysis — unreliable due to device clock skew. |
| `sentAt` | Client SDK | Time the event batch was dispatched to RudderStack. **OR** `sentAt` value manually passed in. | Used in clock-skew correction calculation. **Not recommended** for direct analysis — tainted by device clock skew. |
| `receivedAt` | RudderStack Server | Server-side ingestion timestamp set by the RudderStack Gateway clock when the event is received. | Primary sort key in warehouse destinations. Use for maximum query performance when chronological ordering is not critical. Chronological order is **not** guaranteed. |
| `timestamp` | Computed / Server | Corrected event time. Calculated as: `receivedAt - (sentAt - originalTimestamp)`. Delivered to all downstream destinations. Can be explicitly set in server-side sources. | **Primary event time** for all analytics. Recommended timestamp when chronological order matters. Used for historical replays. |

### Timestamp Computation Formula

```
timestamp = receivedAt - (sentAt - originalTimestamp)
```

This formula corrects for client-device clock skew by using the difference between `sentAt` and `receivedAt` (assumed to occur within milliseconds of each other) as a proxy for the client clock offset, then applying that offset to `originalTimestamp`.

### originalTimestamp

The `originalTimestamp` tells you when the call was invoked on the client device, or the value of `timestamp` that you manually passed in through a server-side library.

The `originalTimestamp` is **not useful for direct analysis** since it is not always trustworthy — it can be easily adjusted and is affected by device clock skew. Its primary purpose is as an input to the `timestamp` computation formula.

### sentAt

The `sentAt` timestamp specifies the clock time on the client's device when the network request was made to the RudderStack API. For libraries and systems that send batched requests, there can be a significant gap between an event's `originalTimestamp` and its `sentAt`.

Combined with `receivedAt`, RudderStack uses `sentAt` to correct the original `timestamp` in situations where a user's device clock cannot be trusted (mobile phones and browsers). The `sentAt` and `receivedAt` timestamps are assumed to occur at approximately the same time (within a few hundred milliseconds), and therefore the difference represents the user's device clock skew, which is applied to correct the `timestamp`.

> **Mobile SDK Note:** The Swift, Kotlin, and C# mobile libraries now set `sentAt` when the batch is complete and initially dispatched to the RudderStack API. This means the computed `timestamp` aligns more closely with `receivedAt` rather than `originalTimestamp`. For most users who are online when events are sent, this does not significantly impact data. However, for applications that utilize offline mode where events queue for extended periods, the `timestamp` will more closely reflect when RudderStack received the events rather than when they occurred on the device.

The `sentAt` timestamp is **not useful for direct analysis** since it is tainted by the user's clock skew.

### receivedAt

The `receivedAt` timestamp is added to incoming messages as soon as they arrive at the RudderStack Gateway. It is used in combination with `sentAt` to correct clock skew, and also to aid with debugging libraries and systems that deliver events in batches.

The `receivedAt` timestamp is the **primary sort key** in RudderStack's warehouse destinations. Use this for maximum query speed when retrieving data from your warehouse.

**Important:** Chronological order of events is **not** ensured with `receivedAt`. Events received in the same batch or from high-throughput sources may share identical `receivedAt` values.

### timestamp

The `timestamp` field specifies when the data point occurred, corrected for client-device clock skew. This is the timestamp that is passed to all downstream destinations and used for historical replays. It is the recommended timestamp for analysis when chronological ordering matters.

If you are using server-side source libraries, or passing calls directly to the HTTP API endpoint, you can manually set the `timestamp` field. This sets the `originalTimestamp` field of the event. If you use a client-side source in device mode, the library generates `timestamp` and you cannot manually set one directly in the call payload.

RudderStack calculates `timestamp` as:

```
timestamp = receivedAt - (sentAt - originalTimestamp)
```

> **Security Note:** For client-side tracking, it is possible for the client to spoof `originalTimestamp`, which may result in a calculated `timestamp` value set in the future or distant past.

**Segment Parity:** RudderStack uses **identical timestamp computation logic** to Segment. The formula `receivedAt - (sentAt - originalTimestamp)` matches Segment's behavior exactly. All four timestamp fields have the same semantics, the same data types, and the same downstream behavior.

## Timestamp FAQ

### Why are events received with timestamps set in the past or future?

If you are using one of RudderStack's client-side libraries, several factors can cause timestamp discrepancies in your event data:

1. **Manual timestamp overrides in server-side sources** — When a manual timestamp is set in the payload with a date in the past or future, it directly affects the computed `timestamp` value for the event.

2. **Analytics.js retry queue replaying old events** — The retry feature supports offline traffic by queuing events in the browser. These events are sent or retried later when an internet connection is available, keeping the original timestamp intact. This can cause events to appear with past timestamps.

3. **Backgrounded mobile apps batching events while offline** — If a user closes or backgrounds a mobile app, events may be queued within the app. These queued events are not sent until the app is re-opened, potentially hours or days later, leading to timestamp discrepancies.

4. **Inaccurate client device clock settings** — Timestamps can be incorrect if the client's device time is inaccurate, since `originalTimestamp` relies on the client device's clock, which can be manually adjusted or drift over time.

5. **Bot traffic with manipulated timestamps** — Internet bots can send requests with unusual timestamps, either intentionally or due to incorrect settings, leading to discrepancies.

| `timestamp` computation | `receivedAt - (sentAt - originalTimestamp)` | `receivedAt - (sentAt - originalTimestamp)` | ✅ Full Parity |
| `receivedAt` | Server ingestion timestamp. Warehouse sort key. | Server ingestion timestamp. Warehouse sort key. | ✅ Full Parity |
| `sentAt` | Client dispatch timestamp. Used for clock-skew correction. | Client dispatch timestamp. Used for clock-skew correction. | ✅ Full Parity |
| `originalTimestamp` | Client invocation timestamp. Input to `timestamp` formula. | Client invocation timestamp. Input to `timestamp` formula. | ✅ Full Parity |
| `type` | 6 event types: identify, track, page, screen, group, alias | 6 event types: identify, track, page, screen, group, alias | ✅ Full Parity |
| `userId` | Persistent user identifier. At least one of `userId`/`anonymousId` required. | Persistent user identifier. At least one of `userId`/`anonymousId` required. | ✅ Full Parity |
| `version` | API version number (currently `2`) | API version number (currently `2`) | ✅ Full Parity |
| Auto-collected context (Web) | 17 fields via Analytics.js (campaign, page, library, locale, userAgent, userAgentData, timezone, ip) | Same 17-field auto-collection matrix | ✅ Full Parity |
| Auto-collected context (iOS) | 22 fields via Analytics-iOS (app, device, library, ip, locale, network, os, screen, traits, timezone) | Same 22-field auto-collection matrix | ✅ Full Parity |
| Auto-collected context (Android) | 24 fields via Analytics-Android (app, device, library, ip, locale, network, os, screen incl. density, traits, userAgent, timezone, bluetooth) | Same 24-field auto-collection matrix | ✅ Full Parity |

> **Migration Note:** When migrating from Segment to RudderStack, no payload-level changes are required for common fields. The identical field names, types, timestamp computation formula, integrations dictionary semantics, and auto-collection behavior mean that existing SDK configurations and server-side integrations will work without modification.
